The Sankofa CLI is the command-line tool for setting up SDK integrations, verifying configurations, and shipping OTA updates. It works across all platforms Sankofa supports.

## Installation

```bash
npm install -g sankofa-cli
```

**Requirements:** Node.js 18+. For iOS builds: macOS + Xcode. For Android builds: Android SDK + Java 17.

## Quick Start

### Any Platform (Analytics)

```bash
# 1. Set up the project — auto-detects platform, patches native files
sankofa init

# 2. Log in to your Sankofa account
sankofa login

# 3. Verify everything is wired up
sankofa check
```

### React Native (Analytics + Deploy)

```bash
# 1. Set up
sankofa init

# 2. Log in
sankofa login

# 3. Verify
sankofa check

# 4. Ship your first release
sankofa release ios

# 5. Push a JS-only patch
sankofa patch ios
```

## Platform Support

`sankofa init` and `sankofa check` auto-detect the platform from the project root:

| Platform | Detection | What `init` does |
| :--- | :--- | :--- |
| **React Native (Expo)** | `expo` in package.json | Adds config plugin to app.json |
| **React Native (bare)** | `react-native` in package.json | Patches native entry files |
| **Flutter** | `pubspec.yaml` exists | Creates config, shows Dart setup |
| **Web (npm)** | `react`/`next`/`vue`/`vite` in package.json | Creates config, shows import setup |
| **Web (CDN)** | `index.html` exists | Creates config, shows `<script>` setup |
| **iOS (Swift)** | `Package.swift` exists | Creates config, shows SPM setup |
| **Android (Kotlin)** | `app/build.gradle` exists | Creates config, shows Gradle setup |

## Commands

### `sankofa init`

Set up Sankofa in any project. Creates `.sankofa.json`, updates `.gitignore`, and handles platform-specific native setup.

```bash
sankofa init
sankofa init --endpoint https://your-server.com --project-id proj_...
sankofa init --force    # overwrite existing config
```

Idempotent — safe to run multiple times.

### `sankofa check`

Verify your SDK integration is correct. Validates credentials, native wiring, initialization, tracking calls, and server connectivity.

```bash
sankofa check              # all modules for detected platform
sankofa check analytics    # analytics-specific checks
sankofa check deploy       # deploy-specific checks (React Native only)
```

Every failed check shows a fix command. Example:

```
  ANALYTICS
  ✓ SDK installed          sankofa-react-native@0.1.0
  ✓ Sankofa.initialize()   app/_layout.tsx
  ✓ Session Replay         recordSessions configured
  ✓ Screen tracking        useSankofaScreen() found
  ✓ Event tracking         Sankofa.track() found
  ✓ Analytics API access   API key valid and accessible

  ✓ Analytics is fully configured and ready!
```

**Analytics checks** (all platforms): SDK installed, initialization, API key validation, session replay, screen tracking, event tracking, user identification, server verification.

**Deploy checks** (React Native): Native bundle provider, config plugin, `notifyAppReady()`, `checkForUpdate()`, `.gitignore` coverage, API access.

### `sankofa doctor`

Low-level toolchain diagnostics. Checks Node, Xcode, CocoaPods, Java, Android SDK, and server reachability.

```bash
sankofa doctor
```

### `sankofa login`

Authenticate via browser. Creates a Deploy Token for the selected project.

```bash
sankofa login
sankofa login --deploy-token sk_deploy_... --project-id proj_...   # CI/CD
sankofa login --endpoint https://your-server.com                   # self-hosted
```

### `sankofa logout`

Remove stored credentials.

```bash
sankofa logout             # both scopes
sankofa logout --project   # project-scoped only
sankofa logout --global    # global only
```

### `sankofa switch`

Switch to a different project without re-authenticating.

```bash
sankofa switch
```

### `sankofa release`

Build and publish a base release (React Native).

```bash
sankofa release ios
sankofa release android
sankofa release ios --publish --rollout 50 --description "v1.2.0"
```

Every release builds the native app, publishes the OTA update, and produces a signed store binary. Pass `--skip-distribution` for OTA only.

### `sankofa patch`

Ship a JavaScript-only update against an existing base release. No native rebuild required.

```bash
sankofa patch ios
sankofa patch android --publish --mandatory
```

### `sankofa preview`

Download and install a published release on your local simulator/emulator.

```bash
sankofa preview ios
sankofa preview ios --label v1.2.0-patch.3
```

### `sankofa status`

Show all releases for the current project with rollout %, install counts, and status.

```bash
sankofa status
sankofa status --env live --platform ios
```

### `sankofa dist`

Build only the signed store binary without publishing an OTA release.

```bash
sankofa dist ios
sankofa dist android --android-format aab
```

### `sankofa submit`

Upload the signed binary to App Store Connect or Play Console.

```bash
sankofa submit ios --apple-api-key-id ABC --apple-api-issuer UUID
sankofa submit android --google-service-account ~/sa.json --google-track internal
```

### `sankofa upgrade`

Check for and install the latest CLI version.

```bash
sankofa upgrade
```

## Feature flags (Sankofa Switch)

Every flag operation is also a CLI operation. The flag subcommands
authenticate with your **dashboard JWT** (via `sankofa login`), not
the Deploy Token — flag CRUD needs full user + project-role RBAC.

### `sankofa flags list`

```bash
sankofa flags list
sankofa flags list --env test
sankofa flags list --include-archived
```

Returns a table of flag key, kind (`boolean` / `variant`), rollout %,
last updated, and status (`active` / `halted` / `archived`).

### `sankofa flags get`

```bash
sankofa flags get new_checkout
```

Prints the flag's default value, attached rule (rollout %, targeting
predicates, cohort includes), and variant list with weights.

### `sankofa flags create`

Creates a new **boolean** flag. For variant flags, create the flag
here first, then add variants via the dashboard or `sankofa demo seed`
(which uses the variant API internally).

```bash
sankofa flags create new_checkout \
  --description "Rebuilt checkout flow" \
  --default false
```

### `sankofa flags toggle`

Shortcut that upserts the flag's rule with a rollout % while preserving
any existing targeting. Great for the progressive-rollout loop.

```bash
sankofa flags toggle new_checkout 10    # start at 10%
sankofa flags toggle new_checkout 50    # ramp to 50%
sankofa flags toggle new_checkout 100
```

Rollout is bucketed deterministically on `SHA-256(distinct_id + flag.key)`,
so ramps are monotonic — bumping 10% → 30% never reshuffles the first
10%.

### `sankofa flags archive`

```bash
sankofa flags archive new_checkout
```

Archived flags always return their default. SDKs still serve the
cached decision until the next handshake, at which point `reason =
archived`.

### `sankofa flags halt` / `sankofa flags resume`

The kill-switch path. `halt` flips the flag to its default
immediately; `resume` clears the halt and re-evaluates the rule.

```bash
sankofa flags halt new_checkout --reason "spike in errors"
sankofa flags resume new_checkout
```

Halt is also exposed as `POST /api/switch/halt-webhook` (x-api-key
auth) — Sankofa Catch or any external system can flip flags without
dashboard access.

### `sankofa flags scan`

Stale-flag scanner. Walks your checkout for `.getFlag("…")` and
`.getVariant("…")` calls across JS/TS/Dart/Swift/Kotlin, then
cross-references the keys against the server.

```bash
sankofa flags scan
sankofa flags scan --strict    # fail CI on any warning
```

Warnings: referenced-in-code-but-missing-on-server (typos), on-server-
but-unreferenced (dead flags). Errors: archived-on-server but still
branched in code. Pass `--strict` to wire scanner into CI.

## Remote config (Sankofa Config)

### `sankofa config list`

```bash
sankofa config list
sankofa config list --env test
```

Table of key, type, current default value (truncated), version, last
updated.

### `sankofa config get`

```bash
sankofa config get max_upload_mb
```

Prints the current default + rule + latest version number.

### `sankofa config set`

Create or update a config item's default value. Type is required on
create; on an existing key with a matching type, `set` only updates
the default value (type mismatches are rejected — archive the old key
and use a new one for a different shape).

```bash
sankofa config set max_upload_mb int 25 --description "Max file upload in MB"
sankofa config set support_url string "https://support.acme.com"
sankofa config set trial_discount_pct float 0.2
sankofa config set maintenance_banner_enabled bool true
sankofa config set pricing json '{"pro":9.99,"growth":49}'
```

JSON values accept any JSON shape — maps, arrays, scalars. The server
validates the payload against the declared type at write time.

### `sankofa config history`

```bash
sankofa config history max_upload_mb
sankofa config history max_upload_mb --limit 5
```

Append-only version log. Every create, default-value update, rule
save, archive, or rollback appends a row.

### `sankofa config rollback`

Non-destructively revert to a historical version — writes a **new**
version whose snapshot matches the target, rather than deleting
history.

```bash
sankofa config rollback max_upload_mb 3 --note "revert accidental bump"
```

The rollback pulls the target version's value + rule snapshot, opens
a transaction, writes a new `rolled_back` version, and updates the
item's default + rule to match. Rolling back a rollback is the same
operation in reverse.

## Demo fixtures

Provision the canonical demo flags + config items used by every
Sankofa example app (web, react-native, html, ios, android, flutter)
in one shot.

### `sankofa demo seed`

```bash
sankofa demo seed
sankofa demo seed --force           # overwrite existing values
sankofa demo seed --skip-flags      # only seed config items
sankofa demo seed --skip-config     # only seed feature flags
```

Creates:

**6 flags** — `new_home_layout`, `checkout_cta_variant` (variant
A/B/C), `onboarding_v2_rollout`, `ai_summary_kill_switch`,
`ab_pricing_page` (variant A/B), `premium_badge_visible`.

**6 config items** — `support_url` (string), `max_uploads_per_day`
(int), `trial_discount_pct` (float), `maintenance_banner_enabled`
(bool), `pricing_table` (json), `theme_colors` (json).

Idempotent by default — keys that already exist are skipped. Pass
`--force` to overwrite the default values to match the manifest.

<Callout title="Pair with an example app" tone="info">
Seed once, then run any of the six Sankofa example projects — every
UI surface in the examples is wired to these exact keys, so dashboard
edits repaint the apps without code changes. See the
[Demo examples guide](/getting-started/demo-examples).
</Callout>

## Authentication

Credentials are resolved in this order (highest wins):

1. **Environment variables** — `SANKOFA_DEPLOY_TOKEN`, `SANKOFA_PROJECT_ID`, `SANKOFA_ENDPOINT`
2. **Project config** — `.sankofa.json` in the project root
3. **Global config** — `~/.sankofa/credentials.json`

### CI/CD

Skip the browser login with a pre-minted Deploy Token:

```bash
export SANKOFA_DEPLOY_TOKEN=sk_deploy_...
export SANKOFA_PROJECT_ID=proj_...
export SANKOFA_ENDPOINT=https://your-server.com

sankofa patch ios --publish --rollout 25
```

## Environment Variables

| Variable | Purpose |
| :--- | :--- |
| `SANKOFA_DEPLOY_TOKEN` | CI-friendly Deploy auth token |
| `SANKOFA_JWT` | CI-friendly dashboard JWT (used by `flags`, `config`, `demo`) |
| `SANKOFA_PROJECT_ID` | Project ID |
| `SANKOFA_ENDPOINT` | Server URL |
| `SANKOFA_ENVIRONMENT` | `live` or `test` |

<Callout title="Which auth for which command?" tone="info">
**Deploy commands** (`release`, `patch`, `submit`, `dist`, `status`,
`preview`) use `SANKOFA_DEPLOY_TOKEN` — pre-mint one from the
dashboard's "Deploy tokens" panel for CI runs.

**Switch / Config / demo commands** (`flags`, `config`, `demo`) use
`SANKOFA_JWT` because flag and config CRUD require full user auth
and project-role RBAC. Either log in interactively via
`sankofa login`, or issue a long-lived JWT from the dashboard for
CI.
</Callout>
