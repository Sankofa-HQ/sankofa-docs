Sankofa Switch and Sankofa Config launch together as the V1 feature-flag
and remote-config surfaces. Both share the same handshake, targeting, and
cohort primitives that already power Deploy — so rollout semantics are
identical and production-hardened from day one.

## What shipped

### Platform

- **Switch** — boolean flags, A/B/n variant flags with deterministic
  weighting, targeting rules (app version range, OS version range,
  country allow/block, cohort include/exclude, user pick-list), rollout
  %, scheduled toggles (timezone-aware), dependencies (cycles rejected
  at save time), halt webhook + resume, and experiment results with a
  95% Wilson CI (plus Bayesian posteriors + P(variant > control) when
  enabled).
- **Config** — typed remote config (`string | int | float | bool |
  json`), cohort-targeted payloads, append-only version history, and
  non-destructive rollback.
- **Audit log** — every mutation (flag, config, rule, variant,
  schedule, dependency) is captured with actor, action, and a diff-able
  value snapshot. Compliance-ready retention (90d primary, 2y mirrored
  for dashboard search).
- **Unified handshake** — both products merge into the existing
  `/api/v1/handshake` contract. SDKs send `If-None-Match`; the server
  returns `304 Not Modified` with an empty body when nothing changed,
  keeping steady-state cost near zero.

### Dashboard

- New **Switch** product: flags list, detail (Overview / Rule / Variants /
  Schedule / Dependencies / Experiments tabs), halt banner, simulate-
  evaluation panel.
- New **Config** product: items list, detail (Overview / Rule / History
  tabs), version timeline with 1-click restore.
- **Audit log** page — filterable by actor, resource, action, date range.
- **Billing** dashboard now routes `?product=switch` and `?product=config`
  to fully-featured 4-tier plan pickers with Paystack checkout.
- **Onboarding** modules screen lists all five products (Analytics,
  Deploy, Catch, **Switch**, **Config**).

### SDKs

Every SDK ships the same three-method API: `getFlag` / `getVariant` /
`onChange` for Switch, `get<T>` / `getAll` / `onChange` for Config. All
SDKs cache locally (platform-native storage), stale-while-revalidate up to
7 days, and send `If-None-Match` to hit the server's 304 path on refresh.

| Platform | Package / Class | Storage | Build status |
| --- | --- | --- | --- |
| Web | `@sankofa/switch`, `@sankofa/config` | `localStorage` | ✅ tsc |
| React Native | `SankofaSwitch`, `SankofaConfig` | native bridge (`SharedPreferences` / `UserDefaults`) | ✅ tsc |
| Flutter | `SankofaSwitch`, `SankofaConfig` | `shared_preferences` | ✅ dart analyze |
| iOS (Swift) | `SankofaSwitch`, `SankofaRemoteConfig` | `UserDefaults` | ✅ xcodebuild |
| Android (Kotlin) | `SankofaSwitch`, `SankofaRemoteConfig` | `SharedPreferences` | ✅ gradle |

**Naming note:** iOS and Android call the remote-config class
`SankofaRemoteConfig` because `SankofaConfig` is already the init-options
type on those platforms. The wire format + handshake contract are
identical across all five SDKs.

### CLI

```bash
# Feature flags
sankofa flags list
sankofa flags create <key> [--description ...] [--default true|false]
sankofa flags toggle <key> <0-100>
sankofa flags archive <key>
sankofa flags halt <key> [--reason ...]
sankofa flags resume <key>
sankofa flags scan [--strict]     # stale-flag scanner

# Remote config
sankofa config list
sankofa config set <key> <type> <value>
sankofa config get <key>
sankofa config history <key>
sankofa config rollback <key> <version>
```

CLI uses dashboard JWT auth for everything except `flags halt` (SDK
webhook, requires `sk_*` API key). Stale-flag scanner covers
`.js/.ts/.tsx/.dart/.swift/.kt` files with one regex path; cross-references
code against the server's flag list.

## Upgrade path

Existing Deploy customers don't need to do anything — Switch and Config
are opt-in. Activate them from the dashboard's **Modules** screen, then
create your first flag (`sankofa flags create`) or config item
(`sankofa config set`).

Self-hosted installs get the new features in the next server upgrade;
no data migration required.

## Known limitations (V1)

- **Real-time cohort entry** uses a batch materialiser that refreshes
  every 5 minutes. A user entering a cohort like "Churn Risk" sees
  their cohort-targeted config within ~5 min of the next tick. Real-
  time entry ships in V2.
- **Auto-halt integration with Catch** — the webhook endpoint
  (`POST /api/switch/halt-webhook`) is live and CLI-driven today.
  Catch itself ships separately; when it does it'll post the
  documented payload to this endpoint automatically.
- **Flag-kind migration** (boolean ↔ variant) is disallowed mid-life.
  Archive and create a new key with the target shape.
- **Usage counters** (evaluations / fetches per month) are enforced at
  the quota layer — hitting a cap is a dashboard warning; the SDK
  continues serving cached decisions.

## Next up
