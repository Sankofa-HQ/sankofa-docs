The Sankofa repository ships **six example apps** — one per platform
— all sharing the same canonical set of 6 feature flags and 6 remote
config items. Once you seed them from the CLI, every UI surface in
every example app renders against the live dashboard; edit a flag,
reload an example, and the hero / CTA / pricing card / maintenance
banner all repaint without a code change.

This page is the fastest way to see Switch + Config working end-to-end.

## What's in the box

| Example | Location | Lab surface | Also applied to |
| --- | --- | --- | --- |
| Web (Vite + React) | `example/sankofa_example_web/` | `FlagsLabPanel` | Hero, CTA, AI summary, uploads, pricing grid, premium badge, support link, maintenance banner |
| React Native (Expo) | `example/sankofa_example_react_native/` | `Lab` tab | Home screen variant CTA + maintenance banner |
| HTML (CDN scripts) | `example/sankofa_example_html/` | `#sankofa-lab` section | Hero variant, CTA, premium nav badge, pricing grid, footer support link, maintenance banner |
| iOS (SwiftUI) | `example/sankofa_example_ios/` | `Lab` tab (`FlagsLabView`) | Tracking view variant CTA + maintenance banner |
| Android (Kotlin + Compose) | `example/sankofa_example_android/` | `FlagsLabActivity` | Main activity variant CTA + maintenance banner |
| Flutter | `sdks/sankofa_sdk_flutter/example/` | `FlagsLabScreen` via science icon | Event tester variant CTA + maintenance banner |

Every example:

- Declares the same six flag keys + six config keys locally (bundled
  defaults) so it renders before the first handshake.
- Subscribes to each key via `onChange` so dashboard edits repaint
  live.
- Forwards `anon_id` on handshake after `identify()` fires.
- Records exposures (web only today) for experiment math.

## The canonical manifest

These are the exact keys `sankofa demo seed` provisions. Copy them
into your own app verbatim if you want to piggyback on the demo flow.

### Flags

| Key | Kind | Default | Rollout | Use in the examples |
| --- | --- | --- | --- | --- |
| `new_home_layout` | boolean | `false` | 50% | Swap hero between classic + v2 layouts |
| `checkout_cta_variant` | variant | `control` | 100% (34/33/33) | CTA copy + colour (control / blue / red) |
| `onboarding_v2_rollout` | boolean | `false` | 25% | Progressive rollout gate for uploader |
| `ai_summary_kill_switch` | boolean | `false` | — | Halt-webhook target — flips AI summary card into paused state |
| `ab_pricing_page` | variant | `A` | 100% (50/50) | Reorders pricing tier cards |
| `premium_badge_visible` | boolean | `true` | 100% | Sparkly Premium badge on identified users |

### Config

| Key | Type | Default | Use |
| --- | --- | --- | --- |
| `support_url` | string | `https://support.sankofa.dev` | Support link in footer + lab |
| `max_uploads_per_day` | int | `25` | Daily upload ceiling shown to user |
| `trial_discount_pct` | float | `0.2` | Discount multiplier on pricing |
| `maintenance_banner_enabled` | bool | `false` | Amber maintenance banner |
| `pricing_table` | json | 3-tier array | Pricing cards content |
| `theme_colors` | json | `{primary, accent}` | Primary + accent hex tokens |

## What to edit first

Five quick edits that demonstrate the full range:

1. **Flip `maintenance_banner_enabled` → `true`** — every example
   shows an amber maintenance banner on reload.
2. **Change `checkout_cta_variant` variants' weights** to `control=0,
   blue=100, red=0` — every example's CTA turns blue and the label
   changes to "Try it free".
3. **Halt `ai_summary_kill_switch`** via `sankofa flags halt
   ai_summary_kill_switch --reason "demo"` — the AI summary card in
   every example flips to the paused state, and the reason tag on the
   flag's Lab row changes to `halted`.
4. **Edit `theme_colors.primary`** to a different hex — the CTA
   buttons and pricing card accent recolour across every example.
5. **Bump `pricing_table`** — replace one tier's price and watch every
   example's pricing grid update simultaneously.

## Cleanup

To remove the seeded keys when you're done experimenting:

```bash
# Archives the flags (SDKs return defaults)
for key in new_home_layout checkout_cta_variant onboarding_v2_rollout \
           ai_summary_kill_switch ab_pricing_page premium_badge_visible; do
    sankofa flags archive "$key"
done

# Config archiving ships with the dashboard detail page's Archive
# button; there's no CLI shortcut yet.
```

Or, if you're working on a throwaway demo project, just delete the
project in the dashboard — everything scoped to it goes with it.

## Extending your own app

The demo manifest isn't sacred — use whatever keys fit your product.
Every example ships a `sankofaDemo.(ts|swift|kt|dart)` file defining
the shared keys + local defaults; copy that file into your app and
swap the keys. The three-step pattern (install → construct before
init → call) is the same.

For the server-side creation side of things, either:

- Use the dashboard Switch / Config UIs.
- Use the CLI (`sankofa flags create`, `sankofa config set`).

The [Switch + Config integration guide](/usage/switch-config-integration)
has the full per-SDK walkthrough.
